Growth Curve — Operations vs Input Size

Assuming sparse graph: E ≈ 2V. Prim with binary heap = O(E log V) ≈ O(2V · log V). For dense graphs (E ≈ V²), Prim still performs well with Fibonacci heap: O(V² + V log V).

Worst Case Analysis

When Does Prim's Perform Worst?

The worst case for Prim's with a binary heap occurs when every edge relaxation causes a decrease-key operation, maximizing the total number of heap operations. This happens when the graph is dense (E ≈ V²) and every new vertex added to the MST triggers updates to many neighboring vertices.

Worst Case: Dense graph with V² edges → O(V² log V) operations. Each of the V extract-min operations costs O(log V), and each of the E decrease-key operations also costs O(log V). Total: O((V + E) log V) = O(E log V).
Why "Worst Case" preset has equal weights: When all weights are equal, Prim's still runs identically — it performs the same number of comparisons but gains no benefit from the priority queue's ordering. The algorithm's behavior is most predictable and thus demonstrably worst-case in operation count.

CLO-3 · PI 1.1 · Greedy Paradigm

How Prim's Uses the Greedy Strategy

A greedy algorithm makes the locally optimal choice at each step, hoping to reach a globally optimal solution. Prim's MST is a perfect example: at each step, it selects the minimum weight edge that connects the current MST to an unvisited vertex — always the cheapest available crossing edge.

Greedy Choice Property: The minimum weight crossing edge between a tree vertex and a non-tree vertex is always safe to add to the MST (proven by the Cut Property).
Optimal Substructure: Any MST of a subgraph is a subtree of the full MST. This means greedy choices accumulate into a globally optimal solution.

Prim's Algorithm

  • Grows MST from a single vertex
  • Always adds cheapest crossing edge
  • Uses a priority queue (min-heap)
  • Better for dense graphs
  • Vertex-centric approach
  • O(E log V) with binary heap

Kruskal's Algorithm

  • Starts with all vertices as singletons
  • Always adds cheapest global edge
  • Uses Union-Find data structure
  • Better for sparse graphs
  • Edge-centric approach
  • O(E log E) sorting dominated
Counterexample: Why Greedy Fails for Shortest Path

Prim's greedy approach works for MST because of the Cut Property — minimum crossing edges are always safe. However, the same greedy strategy does NOT work for shortest paths without Dijkstra's relaxation. Consider the graph below:

A → B: weight 1 (greedy picks this first) | B → C: weight 10 | A → C: weight 3
MST: {A-B(1), A-C(3)} ✓ Greedy choice A-B is correct.
Shortest A→C: Direct A-C(3) ✓ — but naïve greedy via B would give 1+10=11 ✗
This shows greedy works for MST (Cut Property) but needs extra structure for shortest paths.

Greedy Steps Animated

Each iteration of Prim's loop IS the greedy choice. The algorithm never backtracks — once an edge is added to the MST, it stays. This is the essence of greedy: commit to the best local choice and move on.

Greedy Step 1
Start at vertex s. Set key[s]=0, all others=∞. Extract minimum key vertex (s). Mark s as in MST.
Greedy Choice at Each Iteration
Extract vertex u with minimum key (cheapest edge to MST). For each neighbor v not in MST: if edge(u,v) weight < key[v], update key[v] = weight and parent[v] = u.
Proof of Correctness (Cut Property)
For any cut (S, V-S) of the graph, the minimum weight crossing edge is always safe to add to the MST. Prim's always adds a minimum crossing edge → always correct.

📖 What is Prim's Algorithm?

▼

Prim's algorithm (Robert C. Prim, 1957) finds the Minimum Spanning Tree (MST) of a connected, undirected, weighted graph. The MST is a subgraph that connects all vertices with the minimum total edge weight, containing exactly V-1 edges for V vertices and no cycles.

Key Idea: Grow the MST one vertex at a time. Start from any vertex, and repeatedly add the cheapest edge connecting the current MST to a new vertex.

Unlike Kruskal's algorithm which considers all edges globally, Prim's maintains a "frontier" — the boundary between vertices already in the MST and those not yet included. It always picks the cheapest edge crossing this frontier.

  • Works on connected, undirected, weighted graphs
  • Produces a spanning tree with minimum total weight
  • The result is always a tree (V vertices, V-1 edges, no cycles)
  • If multiple edges have the same weight, there may be multiple valid MSTs

🔧 How the Algorithm Works

▼

Prim's algorithm uses a priority queue (min-heap) to efficiently find the minimum weight crossing edge at each step.

Data Structures:
• key[]: minimum weight edge connecting v to MST (initially ∞, 0 for start)
• parent[]: which MST vertex connects to v (initially NIL)
• inMST[]: whether vertex v is in the MST (initially false)
• Priority Queue Q: vertices ordered by key value

Algorithm Steps:

  1. Initialize: key[start]=0, all other keys=∞, all inMST=false
  2. While Q is not empty: Extract vertex u with minimum key
  3. Mark u as inMST[u]=true
  4. For each neighbor v of u: If v not in MST AND w(u,v) < key[v]: Update key[v]=w(u,v), parent[v]=u
  5. MST consists of all edges (parent[v], v) for v ≠ start
⚠ Common Mistake: Confusing Prim's with Dijkstra's. Dijkstra finds shortest paths from a source (cumulative distances). Prim's finds the MST (each edge's own weight, not cumulative). The key[] in Prim's stores the edge weight, NOT the total path distance.

📝 Fully Worked Example

▼

Graph: A-B(2), A-C(3), B-C(1), B-D(4), C-D(5). Start at A.

Initial State
key = [A:0, B:∞, C:∞, D:∞] | parent = [A:-, B:NIL, C:NIL, D:NIL] | Q = {A:0, B:∞, C:∞, D:∞}
Step 1: Extract A (key=0)
Neighbors of A: B(w=2), C(w=3)
B not in MST, 2 < ∞ → key[B]=2, parent[B]=A ✓
C not in MST, 3 < ∞ → key[C]=3, parent[C]=A ✓
Q = {B:2, C:3, D:∞}
Step 2: Extract B (key=2) — Greedy Choice!
Neighbors of B: A(in MST, skip), C(w=1), D(w=4)
C not in MST, 1 < 3 → key[C]=1, parent[C]=B ✓ (decrease-key!)
D not in MST, 4 < ∞ → key[D]=4, parent[D]=B ✓
Q = {C:1, D:4}
Step 3: Extract C (key=1) — Greedy Choice!
Neighbors of C: A(in MST), B(in MST), D(w=5)
D not in MST, 5 > 4 → no update
Q = {D:4}
Step 4: Extract D (key=4) — Done!
No unvisited neighbors. Q is empty.
MST Edges: A-B(2), B-C(1), B-D(4). Total Cost = 7
Result: MST = {A-B(2), B-C(1), B-D(4)}, Total Weight = 7. Note how B-C was updated from 3 to 1 when B was processed (decrease-key operation).

💡 Key Concepts & Terminology

▼
Minimum Spanning Tree: A tree that connects all V vertices using exactly V-1 edges, with minimum total weight. If the graph has unique edge weights, the MST is unique.
Extract-Min: Remove and return the vertex with the smallest key from the priority queue. O(log V) with binary heap.
Decrease-Key: Update a vertex's key to a smaller value. O(log V) with binary heap, O(1) amortized with Fibonacci heap.
Cut Property: For any partition of vertices into sets S and V-S, the minimum weight edge crossing the cut is always in some MST. This is what proves Prim's greedy approach is correct.
Crossing Edge: An edge (u,v) where u is in the current MST and v is not yet in the MST.
⚠ Prim's ≠ Dijkstra's: Prim's key = edge weight to MST. Dijkstra's key = total path distance from source. Using Dijkstra's logic would give wrong MST results!
⚠ Cycles: We skip neighbors already in the MST to prevent cycles. If we didn't check inMST, we'd add back-edges creating cycles.

▼
